1. What the POC Does Today

Engram Compute (Your Containers)

  • API: FastAPI gateway (chat, memory APIs, ETL ingest endpoint)
  • Worker: Temporal worker (activities: enrich, reason, validate, persist)
  • Temporal Server: Workflow engine (self-hosted in your tenant)
  • Temporal UI: Workflow inspection UI (self-hosted)

External Managed Services

  • Zep Cloud: Configured via ZEP_API_URL=https://api.getzep.com/api/v2 (key in Key Vault). This is why your 4 container apps do not include a "zep" container.
  • Unstructured Hosted: Optional â€” POC can call hosted Unstructured, or parse locally via the Python library.
  • LangSmith: Optional engineering observability/evals â€” not required for runtime.

ğŸ“Œ Where Does Ingested Data Go?

By default: Zep, not Postgres. Engram's ingestion endpoint extracts text and indexes chunks as Zep "facts." Unstructured is the parser/chunker; Zep is the destination. Postgres is used for Temporal persistence and reserved for future platform data.

2. Enterprise Deployment

The enterprise "Engram Engine" goal: everything self-hostable is self-hosted in the customer tenant, with vendor agreements for any managed/SaaS dependencies.

Enterprise Core (Customer Tenant)

  • Engram API: Kubernetes deployment
  • Engram Worker: Kubernetes deployment
  • Temporal: Self-hosted in K8s/ACA (or Temporal Cloud)
  • Zep: Self-hosted in K8s/ACA
  • Postgres: Azure Postgres Flexible Server with encryption + backups (or in-cluster with PV)
  • Object Storage: Azure Blob / S3 for raw docs + parsed artifacts
