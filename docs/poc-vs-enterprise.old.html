POC vs Enterprise Deployment: What runs where

Goal: make it unambiguous what you run in a low-cost POC vs what you run for a fully supported enterprise tenant deployment.

Status: Draft (2025-12-13). Links: Pricing, Test Plan, Success Metrics.

1) What the POC is doing today (current architecture)

Engram compute (your containers):

  • API: FastAPI gateway (chat, memory APIs, ETL ingest endpoint)
  • Worker: Temporal worker (activities: enrich, reason, validate, persist)
  • Temporal Server: workflow engine (self-hosted in your tenant)
  • Temporal UI: workflow inspection UI (self-hosted in your tenant)

External managed services (POC):

  • Zep: currently configured as Zep Cloud via ZEP_API_URL=https://api.getzep.com/api/v2 (key stored in Key Vault in IaC). This is why your 4 container apps do not include a “zep” container.
  • Unstructured hosted: optional (POC can call hosted Unstructured, but today the code path can also parse locally via the Unstructured Python library).
  • LangSmith (LangChain Plus): optional (engineering observability/evals), not required for runtime.

1.1 Does Unstructured output go to Postgres today?

No—by default it goes to Zep. Engram’s ingestion endpoint extracts text and indexes chunks as Zep “facts”.

  • Unstructured is the parser/partitioner/chunker.
  • Zep is the destination memory store for the extracted chunks (facts + episodic sessions).
  • Your Azure Postgres Flexible Server is currently used for Temporal persistence and reserved for future platform data, but it is not the default destination for ingested chunk content unless you explicitly implement that path.

2) What a fully supported enterprise tenant deployment looks like

The enterprise “Engram Engine” goal is: everything self-hostable is self-hosted in the customer tenant, with vendor agreements in place for any managed/SaaS dependencies.

2.1 Enterprise core (customer tenant)

  • Engram API (Kubernetes deployment)
  • Engram Worker (Kubernetes deployment)
  • Temporal (self-hosted in K8s/ACA)
  • Zep (self-hosted in K8s/ACA)
  • Postgres with encryption + backups (Azure Postgres Flexible Server or in-cluster Postgres w/ PV)
  • Object storage for raw docs + parsed artifacts (Azure Blob / S3 / customer choice)

2.2 Why Kubernetes + PV

  • Persistence: Zep/Temporal/Postgres require durable state; PVs provide the storage substrate.
  • Isolation: customer tenant boundaries + network policy controls.
  • Supportability: backups, restore drills, upgrades, and predictable operating model.

3) Naming + environments (dev/test/uat/prod)

Current naming uses “staging-*” for the POC. That’s fine. For enterprise readiness we will standardize parameterized IaC that can create:

  • dev: fast iteration, smaller quotas
  • test: integration tests, synthetic load
  • uat: customer acceptance, governance rehearsal
  • prod: SLO-backed, audited runtime

See: Enterprise Environment Model (dev/test/uat/prod).
